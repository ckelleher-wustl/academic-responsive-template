Current Projects

Modeling API Learning

Current programming practices rely heavily on the use of APIs (Application Programming Interfaces) and frameworks. However, APIs can be challenging to learn and use. Existing research primarily focuses on specific barriers programmers encounter while learning APIs, providing a fragmented understanding of the process. We are developing models of the API learning process informed by Information Foraging, Cognitive Load Theory, External Memory and Sensemaking. Our initial studies provide preliminary validation of these models through examining the behavior of programmers learning the React JS API. Additionally, these models show how programmers move through the process of API learning and identify barriers they encounter.

    • Code Stories

    Understanding and using an existing codebase can be difficult. Programmers often struggle to answer basic questions: what code influences which aspects of the system? Why is it structured the way it is? Some of this information was observable during program creation but is not stored. Source code commits lump too many changes together, often representing the end state of a problem solving process but removing traces of the process, thereby making it difficult to understand the reasoning for them. Yet, there can also be too much information. Key stroke level histories are detailed but difficult to navigate and extract information from. We are working to create Code Stories that capture the history of how software programs evolved that organize changes based on a problem-solution structure and incorporate code history, output history and web foraging history to capture the influences that shape the code.

    Tools to Support TAs

    Universities increasingly rely on undergraduate TAs to provide accessible help for students in computer science courses. But, we know relatively little about how TAs and learners interact, how those practices compare to the best practices for tutoring, and where TAs could use some extra support. We are studying what occurs during help sessions, identifying barriers, and designing tools to address those barriers. The first such tool focuses on supporting TAs in debugging student code and understanding the bugs that exist in student code.

    Storytelling Alice

    Storytelling Alice used storytelling as a means to the end of engaging middle school girls in computer programming. Rather than presenting programming as an end in itself, it was presented as a means of self-expression through stories created and reified using 3D animation. By designing the environment to support storytelling, it was significantly more successful at engaging young girls in programming. Users of Storytelling Alice spent 42% more time programming and were more than 3 times as likely to sneak extra time to program (51% vs 16%). In the years since Storytelling Alice, storytelling has become a fairly common approach for engaging a more gender-diverse audience in programming.

    • Games for Stroke Rehabilitation

    Roughly one in four of us can expect to experience a stroke at some point in our lives, often leading to long term disability. One of the challenges associated with recovering from a stroke is that each individual experiences a different injury, thus leading to a distinct set of deficits. Individualized rehabilitation is particularly critical. We did some of the early work on the design of personalizable games for stroke rehabilitation, helping to explore how to design games with a physical rehabilitation goal and how those games impacted individuals with stroke. Along the way, we developed novel strategies for calibrating games to account for changing daily needs as well as strategies for in-game reward systems.

    • Supporting Novice Code Reuse

    Formal opportunities to explore computer science are still relatively rare for young students. Consequently, it is important that programming environments for kids enable them to learn from each others' code. We have explored a variety of interfaces that record execution histories and allow users to interactively explore them to identify the behavior associated with each individual action or groups of actions. These tools also allow users to select action of interests to reuse in another program and guide the process of integrating selected code into that target program.

    • Generating and Presenting Tutorials

    Formal opportunities to explore computer science are still relatively rare for young students. Consequently, it is important that programming environments for kids enable them to learn from each others' code. We have researched how novice programmers perform code localization and used that to design interfaces that 1) allow novices to select, adapt, and reuse code others have written within their own projects and 2) enable novices to explore and understand the behavior of code they did not write.

    • Code Puzzles and Cognitive Load

    We explored the use of code puzzles similar to Parsons puzzles as a way to help kids learn to program. Our research contributed insights about how to design effective puzzles and the impact of distractor elements (code elements that are not part of the correct solution). We then created a model to predict the Cognitive Load a learner would experience from a selected puzzle, given their past history. Analysis of the underlying learned model revealed concepts that behaviors that were predictive of success.

    • Static Code Suggestions

    Sometimes the most difficult questions are the ones you don't realize that you should be asking. In this project, we explored using static code analysis of programmers' code as they were working to suggest methods and programming constructs that might be relevant to them. These were presented in a side panel of options. Our results suggest that this is a potentially valuable form of support that is currently rare in systems. Novice programmers chose to engage with these suggested items more frequently than any other form of support available, including documentation, tutorials, and code puzzles. Further, suggestions appealed to a substantially broader proportion of users than other forms of support: 70% of users used suggestions vs 27.5% for the next most used form.
